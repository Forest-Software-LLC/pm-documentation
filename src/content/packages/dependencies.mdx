# Dependencies & Versioning

This page defines exactly how Forest records, resolves, and versions dependencies.

## How dependencies are recorded

Dependencies are read from your `forest.json`, nothing else. `forest publish` does **not** scan your source for `require` calls:

1. `forest i @scope/name` resolves the package, writes it into the `dependencies` object of `forest.json` (as `^<installed-version>`), and updates `forest-lock.json`.
2. `forest publish` uploads the `dependencies` object exactly as it appears in `forest.json`.

So "automatic dependency detection" at publish time means: whatever you installed (and didn't remove) is a dependency. If you hand-edit `forest.json` to add or change an entry, that's honored too — run `forest install` afterward to re-resolve.

## Version ranges

Dependency versions are **ranges, not exact pins**, following [SemVer](https://semver.org). The default written by `forest i` is a caret range:

| Syntax | Meaning |
|---|---|
| `^1.2.3` | ≥ 1.2.3 and < 2.0.0 — compatible updates (the default) |
| `~1.2.3` | ≥ 1.2.3 and < 1.3.0 — patch updates only |
| `=1.2.3` | Exactly 1.2.3 |
| `>=1.2, <1.5` | Comma-separated comparators, all must hold |
| `1.2.*` | Any patch of 1.2 |

<Callout type="warning">A bare version like `"1.2.3"` is treated as `^1.2.3` (caret semantics), **not** an exact pin. Use `=1.2.3` if you need to pin exactly.</Callout>

Exact ranges (`=`) are how a set of related packages can be kept in lockstep — though if two packages must always move together, they usually want to be [one package with one root](/packages/server-client) instead.

Actual installed versions are pinned by `forest-lock.json`; ranges only matter when the lockfile is (re)generated.

## Conflict resolution: dedupe first, side-by-side when necessary

When the same package appears more than once in the dependency graph:

- **Compatible ranges are deduplicated.** If every requirement on `@forest/signal` can be satisfied by one version, exactly one copy is installed and shared.
- **Incompatible ranges install side by side.** If package A needs `^1.0.0` and package B needs `^2.0.0`, both versions are installed, each nested where its dependent can reach it. Nothing fails — but you pay in duplicate code, and instances of "the same" class from the two copies won't be shared, so prefer depending on compatible ranges when you can.

On disk, shared and direct dependencies live flat in `packages/`, while version-conflicted or transitive-only packages nest inside their dependent's own `packages/` folder. Forest generates small pointer modules (`return require(script.Parent…)`) so every package finds its dependencies at a predictable relative location. All links are **relative instance paths**, which is why the whole `packages/` tree can be mounted anywhere in the datamodel — see [Using Packages](/packages/usage#where-packages-live).

## Circular dependencies

Circular dependencies are **not rejected** — neither at publish nor at install time, and the resolver handles them without looping. In practice a cycle can only be created retroactively: you can only depend on versions that are already published, so `A → B → A` requires publishing A, then B depending on A, then a *new* version of A depending on B.

Treat cycles as a design smell rather than a tool feature: if two packages genuinely need each other, they are one package — give it a single root that exposes both surfaces (see [Recipe: Server & Client APIs](/packages/server-client)).

## Versioning your own package

`forest publish` manages your version number for you. It asks what kind of change you made (bugfix / new feature / breaking change) and bumps patch, minor, or major accordingly — plus a follow-up question to catch accidental breaking changes. You can enter a version manually, but this is discouraged: consumers on caret ranges rely on your version numbers meaning what SemVer says they mean.

- First publish defaults to `0.1.0`.
- A version, once published, can never be reused.
- Because consumers default to `^`, a **major** bump is the only bump that existing users won't pick up automatically which is the the expected behavior for breaking changes.
